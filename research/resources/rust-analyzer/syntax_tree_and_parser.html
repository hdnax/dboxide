Syntax Tree & Parser

Official site: Link.

Commit link: 2020-01-09

Architectural Overview

  • The syntax subsystem is divided into three isolated components:
    • rowan: A generic, language-agnostic library for immutable syntax trees (based on the Red-Green tree model).
  • syntax (crate): Wraps rowan to provide a rust-analyzer-specific API. This is the only crate that knows about rowan.
  • parser (crate): A standalone recursive descent parser that outputs events. It knows nothing about the syntax tree structure.

Design Goals

  • Lossless (full fidelity): Preserve all whitespace, comments, and invalid tokens.
  • Semantic-less: Purely structural; no type checking, name resolution, or hygiene attached.
  • Context-free: Syntax trees are simple value types that can exist without external context (like a database or compiler session).
  • Resilient: Always produce a tree, even for invalid input, which means syntax nodes may be partial.
  • Performance: Prioritize memory/CPU efficiency (uses unsafe for optimizations).
  • Decoupled: The parser logic and the syntax tree structure are isolated, allowing them to evolve independently.
  • Intuitive traversal: The API should support easy navigation (parent, children, siblings).

The Three-Layer Syntax Tree

The core data structure is a “Red-Green Tree” (inspired by C#’s Roslyn compiler). It splits the tree into three distinct layers to balance memory usage with ergonomic API access.

  • Syntax trees are semi-transient: They do not always exist in memory for all files. They are lowered to a lossy but compact representation. However, they can be converted back to a syntax tree.

Layer 1: GreenNode (The Storage)

  • GreenNode is the actual data storage.

    • Purely functional.
    • Have an arbitrary arity.
    • Immutable.
    • Persistent.

  • Conceptual (unoptimized) example - from rust-analyzer:

    #![allow(unused)]
fn main() {
// Runtime tag for a Node/Token
// Nodes and tokens both use the same tag kind?
#[derive(PartialEq, Eq, Clone, Copy)]
struct SyntaxKind(u16);

#[derive(PartialEq, Eq, Clone)]
struct Node {
    kind: SyntaxKind,
    // Only the byte-length is stored, offset is not stored
    text_len: usize,
    // A generic node with a sequence of children nodes and tokens
    // `Arc` is used, so it allows for multithreading
    children: Vec<Arc<Either<Node, Token>>>,
}

#[derive(PartialEq, Eq, Clone)]
struct Token {
    kind: SyntaxKind,
    text: String,
}
}

  • In this sense of representation, Token is a type of leaf Node, while a Node itself is an interior Node.

  • Notable points:

    • Only one generic node with a generic structure - other than that, tree nodes are untyped (with a runtime type tag).
    • The original text is the aggregation of all children’s texts.
    • To access a child of a particular type, a linear traversal of the children is generally required.
    • The maximum cost of modifying a tree grows linearly with the tree’s height.
    • An invalid input would be wrapped an ERROR node.
    • Errors are not incorporated into the syntax tree.

<------------------------- Single Heap Allocation ------------------------->
  • Idea:
    • A standard struct with a Vec of children would require at least two separate allocations: one for the node, one for the vector’s buffer.
    • Use a dynamically sized type (DST) for GreenNode.
  • Mechanism:
    • The node header (ref count, kind, length) and the array of children are stored contiguously in a single memory block.
    • Instead of Arc<Either<Node, Token>>, a single pointer is used with a tag in the last bit to differentiate between Node and Token (pointers to dynamically allocated memory are typically word-aligned so the last bit is unused anyways).
  • Interning: Tokens are reused instead of being allocated on the heap every time: 1 + 1 would contain only one token for 1. The interned result is an Arc<Node> instead of an index to the interning table, so that the syntax tree is self-contained.
  • The length of the text is represented by the type TextSize, which is an opaque (newtyped) u32.
  • SmolStr: String wrapper that stores small strings (like keywords fn, if) inline on the stack (without heap allocation).
    • Note: The text suggests moving this data directly into the interned token allocation in the future.

Alternative Designs

Dealing With Trivia

  • Explicit nodes (Rowan/IntelliJ): Trivia (whitespace) exists as a standard sibling node, creating a “lossless” concrete tree that is easy to modify but “noisy” to traverse.

  • Attached trivia (Roslyn/Swift): Trivia is hidden as “leading/trailing” properties inside tokens, ensuring the tree structure contains only logical elements.

  • Linked list (Dart): A hybrid approach that maintains a clean semantic tree alongside a parallel, doubly-linked list of all tokens to separate logic from formatting.

  • Remarks:

    • There is no standard approach to this.
    • @dbml/parse uses the “attached trivia” approach.

Accessing Children
  • Linear Scan (IntelliJ/Rowan): Stores children in a dynamic list, including trivia. To find a specific node (e.g., function body), you must iterate past all preceding siblings, resulting in O(N) access but minimal memory overhead since missing nodes take zero space.
  • Fixed Slots (Roslyn/Swift): Assigns a permanent index to every possible child defined by the grammar (e.g., “Return Type” is always at index 7). This grants O(1) random access but increases memory usage by allocating slots for optional nodes that aren’t present (None) and requires removing trivia from the tree structure to maintain index stability.

Mutable Trees
  • Mutable trees (IntelliJ): Allow Imperative editing.
    • You locate a node and modify it directly in-place (node.delete()).
    • This offers a simple, intuitive API but introduces complex thread-safety issues (requires read/write locks) since the data changes under the feet of other readers.
  • Immutable trees (Rust-analyzer):** Allow **Functional** editing.
    • You cannot change a node; you must use a Builder or Rewriter to schedule changes and generate a brand new tree (path copying).
    • This makes the API verbose (“annoying”) but trivializes concurrency, as background threads can safely analyze old snapshots while edits happen on a new one.
#![allow(unused)]
fn main() {
// MUTABLE (IntelliJ Style)
// Direct, in-place modification. Simple but requires locking.
for bound in bounds {
    // The tree changes immediately
    bound.delete();
}

// IMMUTABLE (rust-analyzer Style)
// Indirect. Must collect edits and rebuild the tree structure.
let mut builder = tree.edit();
for bound in bounds {
    // Schedule deletion on the 'builder', not the node itself
    builder.delete(bound);
}
// Apply generates a completely new root node
let new_tree_root = builder.apply();

}

Layer 2: SyntaxNode (The Cursor / RedNode)

  • The GreenNode API is a bit inconvenient:

    • No parent pointer.
    • Identity issue: Reused green nodes from different parts of the trees are treated equally.

  • SyntaxNode/RedNode (also called cursors/zippers): Add parent pointers.to green nodes.

  • Conceptual (unoptimized) example from rust-analyzer:

    #![allow(unused)]
fn main() {
type SyntaxNode = Arc<SyntaxData>;

struct SyntaxData {
    offset: usize,
    parent: Option<SyntaxNode>,
    green: Arc<GreenNode>,
}

impl SyntaxNode {
    fn new_root(root: Arc<GreenNode>) -> SyntaxNode {
        Arc::new(SyntaxData {
            offset: 0,
            parent: None,
            green: root,
        })
    }
    fn parent(&self) -> Option<SyntaxNode> {
        self.parent.clone()
    }
    fn children(&self) -> impl Iterator<Item = SyntaxNode> {
        let mut offset = self.offset;
        self.green.children().map(|green_child| {
            let child_offset = offset;
            offset += green_child.text_len;
            Arc::new(SyntaxData {
                offset: child_offset,
                parent: Some(Arc::clone(self)),
                green: Arc::clone(green_child),
            })
        })
    }
}

impl PartialEq for SyntaxNode {
    fn eq(&self, other: &SyntaxNode) -> bool {
        self.offset == other.offset
            && Arc::ptr_eq(&self.green, &other.green)
    }
}
}

Optimizations

  • Tree traversal is common and should be optimized.
    • Atomic avoidance (Rc over Arc): Traversals are strictly single-threaded, so rowan uses Rc (non-atomic reference counting) for parent pointers. This eliminates the CPU overhead of atomic synchronization during tree walking.
    • Thread movement: Since SyntaxNode is !Sync, you cannot send it across threads. Instead, you send a “coordinate”: (GreenNode, TextRange) or lighter (FileId, TextRange). The receiving thread “hydrates” this into a real node by traversing from the root to that range.
  • Transient trees: rust-analyzer rarely stores full trees (GreenNode). It prefers storing the (FileId, TextRange) and reparsing on demand.
    • Pros: Prevent memory exhaustion from holding thousands of syntax trees.
    • Cons: Trade-off CPU.
  • Root-only Arc:
    • Only the root SyntaxNode holds the expensive Arc<GreenNode>, serving as the single anchor for the entire tree’s memory.
    • All descendant nodes hold a raw pointer to their data (for speed) and an owning Rc to their parent (for safety).
    • Explanation: As long as a child node is alive, its Rc keeps the parent alive, bubbling up to the root, which keeps the Arc alive, ensuring the raw pointers never dangle.
    • Result: Traversing the tree requires only fast, non-atomic Rc increments, incurring the cost of an atomic Arc bump only once (when creating the root).
  • Object pooling:
    • Since SyntaxNode is !Sync (single-threaded), rowan maintains a thread-local “free list” of unused node objects.
    • Instead of asking the OS for memory (malloc) for every node visited, the traversal simply pops an existing object from the pool and overwrites its fields.
    • The pool size only needs to match the depth of the traversal, meaning a few dozen pre-allocated slots can handle massive trees without a single heap allocation during the walk.
    • This allows the API to yield “owned” nodes (which are easier to use than references) for the negligible cost of a pointer move.

Alternative Designs

Memoized RedNodes
  • Memoized nodes (C#/Swift): Each SyntaxNode is a heavy, heap-allocated object (Arc) that permanently stores its offset, parent, and a cached list of all children.
    • This enables true pointer equality (checking if two nodes are the same is instant) but doubles memory usage because the entire tree is duplicated in the “Red” layer. C# uses weak references to try to reclaim this space.
  • Cursor approach (rust-analyzer): Reject memoization in favor of “Cursors”—transient views that calculate positions on the fly. This keeps memory usage proportional to traversal depth (stack-like) rather than the total size of the syntax tree.
#![allow(unused)]
fn main() {
// Memoized Design (C# / Swift)
// Persistent, heavy wrapper around the data.
struct SyntaxData {
    offset: usize,
    parent: Option<SyntaxNode>, // Back-link
    // Caches children forever. High memory cost.
    children: Vec<OnceCell<SyntaxNode>>,
}

}

Layer 3: AST (The API)

  • A strongly typed API offers more convenience for the user compared to GreenNodes, as they are untyped (but they are naturally suitable for error nodes and trivial nodes).

  • Example: A class syntax node would be more convenient if it is typed with well-known methods.

  • Example AstNode trait (from rust-analyzer):

    #![allow(unused)]
fn main() {
pub trait AstNode {
  fn cast(syntax: SyntaxNode) -> Option<Self>
  where
      Self: Sized;

  fn syntax(&self) -> &SyntaxNode;
}
}

  • The classes are auto-generated. One example from rust-analyzer:

    #![allow(unused)]
fn main() {
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct FnDef {
    syntax: SyntaxNode,
}

impl AstNode for FnDef {
    fn cast(syntax: SyntaxNode) -> Option<Self> {
        match syntax.kind {
            SyntaxKind::FN => Some(FnDef { syntax }),
            _ => None,
        }
    }
    fn syntax(&self) -> &SyntaxNode {
        &self.syntax
    }
}

impl FnDef {
    pub fn param_list(&self) -> Option<ParamList> {
        self.syntax.children().find_map(ParamList::cast)
    }
    pub fn ret_type(&self) -> Option<RetType> {
        self.syntax.children().find_map(RetType::cast)
    }
    pub fn body(&self) -> Option<BlockExpr> {
        self.syntax.children().find_map(BlockExpr::cast)
    }
    // ...
}
}

  • Variants such as expressions are represented as enums. One example from rust-analyzer:

    #![allow(unused)]
fn main() {
#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub enum AssocItem {
    FnDef(FnDef),
    TypeAliasDef(TypeAliasDef),
    ConstDef(ConstDef),
}

impl AstNode for AssocItem {
    ...
}
}

  • Shared syntactic elements (like names, loop bodies, or doc comments) are modeled via traits:

    #![allow(unused)]
fn main() {
trait HasName: AstNode {
    fn name(&self) -> Option<Name>;
}

impl HasName for FnDef {
    fn name(&self) -> Option<Name> {
        self.syntax.children().find_map(Name::cast)
    }
}
impl HasName for StructDef {
    fn name(&self) -> Option<Name> {
        self.syntax.children().find_map(Name::cast)
    }
}
}

Alternative Designs

Semantic Full AST
  • IntelliJ’s PSI (Program Structure Interface) is a “rich” AST that abstracts over the origin of the code.
    • A PsiMethod object looks and behaves identically whether it comes from a user’s source file (.java) or a compiled dependency (.jar).
    • This hides the complexity from the plugin developer.
  • Attached semantics: Unlike a pure AST which only describes text, PSI nodes carry logic. They include methods like resolve() or getSuperClass(), allowing direct semantic queries on the tree nodes themselves.
  • To optimize memory, a PSI element is initially backed by a lightweight “stub” (serialized metadata from indices). It transparently inflates into a full, heavy AST only when the user actually opens the file or accesses the method body.
